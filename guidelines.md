# Development Guidelines — obd2-mcp-server

## Project Overview

This is a Model Context Protocol (MCP) server written in Python that exposes OBD-II diagnostic data to GitHub Copilot via FastMCP. The server communicates over stdio and uses a local SQLite database.

## Tech Constraints

- **Python 3.11+** — use modern syntax (match/case, type hints with `|` union, etc.).
- **uv** — all commands run through `uv`. Never use `pip install` directly. Add dependencies with `uv add <package>`.
- **FastMCP** — use the `@mcp.tool()` decorator to register tools. Do not use the low-level MCP protocol classes directly.
- **SQLite** — all data lives in `obd2.db`. No ORM; use raw SQL via the `sqlite3` stdlib module.

## Code Style

- Format with `ruff format`. Lint with `ruff check`.
- Use type hints on all function signatures.
- Keep `main.py` lean — it should only contain tool definitions and the server entry point. Factor helpers into separate modules when they exceed ~30 lines.
- Prefer f-strings over `.format()` or `%` formatting.
- Use `pathlib.Path` for file system paths, not `os.path`.

## Database Conventions

- The database file is `obd2.db` in the project root.
- Table and column names use `snake_case`.
- Always use parameterized queries (`?` placeholders) — never interpolate user input into SQL strings.
- Wrap database access in context managers (`with sqlite3.connect(...) as conn:`).
- `seed.py` is the single source of truth for the schema and initial data. Running it should be idempotent (use `CREATE TABLE IF NOT EXISTS` and `INSERT OR REPLACE`).
- Use **indexes** on columns used in `WHERE` / filter clauses (e.g., `idx_category`, `idx_severity`). Declare them in `seed.py` alongside the schema.
- All filtering and pagination (`LIMIT`, `OFFSET`, `WHERE severity = ?`) must happen **in SQL**, not as Python post-processing. Never fetch a large result set and then slice or filter in Python.

## MCP Tool Design

- Each tool must have a clear, single-purpose docstring — Copilot uses this to decide when to call the tool.
- Tool parameters should be simple types (`str`, `int`). Avoid complex objects.
- Return plain strings or dicts. FastMCP handles serialization.
- Validate input at the tool boundary (e.g., normalize DTC codes to uppercase, strip whitespace).
- **Clamp `limit` and `offset`** at the tool boundary: `max(min(limit, MAX), 1)` for limit, `max(offset, 0)` for offset. This prevents negative or zero-page queries.
- Return helpful "not found" messages rather than empty results or exceptions.

## Error Handling

- Never let exceptions propagate to the MCP client unhandled. Catch database errors and return user-friendly messages.
- Log errors to stderr (the MCP protocol uses stdout for communication).
- Use Python's `logging` module, not `print()`, for debug output.

## Testing

- Write tests with `pytest`. Run with `uv run pytest`.
- Test tool functions directly by importing them — no need to spin up the full MCP server for unit tests.
- Mock the database connection for unit tests; use a real `:memory:` SQLite database for integration tests.
- Use the `fts_db` fixture when testing FTS5 full-text search behaviour; use `memory_db` for tests that only need the base table (LIKE fallback).
- Include **boundary / parameter validation tests** (negative limit, zero offset, huge values) for every tool that accepts pagination parameters.
- Each new DTC code added to `seed.py` must have a **unique code**. Run a duplicate check before committing new seed data.

## Git Practices

- Commit messages follow conventional commits: `feat:`, `fix:`, `docs:`, `chore:`.
- Keep commits small and focused — one logical change per commit.
- Do not commit `obd2.db` to version control (it is generated by `seed.py`). Add it to `.gitignore`.

## File Organization

| File / Dir         | Purpose                                         |
| ------------------ | ----------------------------------------------- |
| `main.py`          | FastMCP server entry point and tool definitions  |
| `db.py`            | Database access layer (all SQLite queries)       |
| `seed.py`          | Database schema creation and data seeding        |
| `obd2.db`          | SQLite database (gitignored, generated)          |
| `pyproject.toml`   | Project metadata, dependencies, tool config      |
| `.vscode/mcp.json` | Registers the server with VS Code / Copilot      |
| `tests/`           | pytest test suite (`uv run pytest`)              |
| `guidelines.md`    | This file — development rules and conventions    |
| `README.md`        | Project overview, setup, and usage instructions   |
